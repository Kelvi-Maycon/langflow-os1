# Package Selection

## Use Quick Plan When

- the request is narrow
- the work is low risk
- the audience is mostly one person or a small team
- the user wants fast next steps more than formal documentation
- the work can be explained in a short brief plus phased tasks

Quick plan output:

- objective
- assumptions
- recommended approach
- phase outline
- immediate tasks
- top risks
- open questions

## Use Standard Plan When

- the work involves multiple steps or stakeholders
- the user asks for a plan plus tasks
- a lightweight PRD would reduce ambiguity
- the work needs milestone and dependency tracking
- there is enough complexity that scope and non-goals matter

Standard plan output:

- context brief
- goals and non-goals
- stakeholder or user summary
- PRD-lite
- execution plan
- milestone map
- task backlog
- risks, dependencies, metrics
- open questions

## Use Full Plan Package When

- the work is costly, cross-functional, or high visibility
- multiple teams depend on sequencing
- rollout, communication, or change management matters
- the user explicitly asks for a full PRD or operating plan
- failure would be expensive in time, money, or reputation

Full plan package output:

- context brief
- full PRD
- roadmap by phase
- backlog with acceptance criteria
- risk register
- decision log
- rollout plan
- communication plan
- success metrics and review cadence

## Compression Rules

- If the user asks for "just the tasks," still include a compact scope block.
- If the user asks for "just a PRD," still include milestones, risks, and next steps when delivery depends on them.
- If the request is personal planning, translate PRD language into outcome, plan, habits, milestones, and review checkpoints.
